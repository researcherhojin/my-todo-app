# TodoList 전략 정의서

이 문서는 프로젝트의 설계 원칙, 개발 품질 기준, LLM 에이전트 하네스를 정의한다.
새로운 기능을 만들거나 기존 구조를 변경할 때 이 문서의 원칙에 부합하는지 먼저 확인한다.

---

## 1. 프로젝트 목적

**문제**: AI가 만든 TodoList는 기능이 과다하고, 디자인이 획일적이며, "AI가 만든 것 같은" 느낌이 강하다.

**가설**: UX 리서치 기반의 디자인 원칙을 엄격하게 적용하고, Issue→Branch→PR→Merge 워크플로우로 개발하면, 실제 서비스 수준의 결과물을 만들 수 있다.

**핵심 차별점**: 기능의 양이 아니라 UX의 질. Playwright 스크린샷으로 모든 상태를 증명한다.

---

## 2. 설계 원칙

### 2.1 UX 우선 (UX-first)
기능을 추가하기 전에 "이 기능이 사용자에게 꼭 필요한가?"를 먼저 묻는다.
"기능이 많은 앱"이 아니라 "필요한 것만 있는 앱"이 목표다.
**근거**: Things 3 — 전 세계 TodoList 앱 중 기능이 가장 적지만, 평점이 가장 높다.

### 2.2 증거 기반 검증 (Evidence-based)
"동작하는 것 같다"가 아니라 Playwright 스크린샷으로 증명한다.
- pytest 통과 + 브라우저 캡처 = PR 승인 조건
- 모든 PR에 스크린샷 테이블 포함
- **"스크린샷이 없는 PR은 증거가 없는 주장이다"**

### 2.3 기계적 실행 (Mechanical execution)
디자인 규칙은 CLAUDE.md에 정의하고, 코드는 규칙을 실행만 한다.
- 색상 3가지 — 예외 없음
- border-radius 8px — 예외 없음
- **규칙을 바꾸고 싶으면 CLAUDE.md를 수정한다. 코드에 예외 분기를 넣지 않는다.**

### 2.4 느슨한 결합 (Loose coupling)
models → database → routers → main. 역방향 import 금지.
**이유**: database.py를 수정해도 models.py에 영향이 없어야 한다.

## 3. LLM 에이전트 하네스 (Harness Engineering)

이 프로젝트는 LLM(Claude Code)이 주요 개발 도구다.
LLM은 강력하지만 체계적으로 실패하는 패턴이 있다.

### 3.1 할루시네이션 방어 (Hallucination)
LLM은 존재하지 않는 함수, 파라미터를 자신 있게 사용한다.

**방어:**
- 함수를 호출하기 전에 시그니처를 읽는다: grep -n "def function_name" app/*.py
- 패치 대상이 모듈 레벨인지 local import인지 확인한다
- "아마 이럴 것이다"로 코드를 쓰지 않는다. 모르면 먼저 읽는다

### 3.2 확증 편향 방어 (Context Length Bias)
컨텍스트가 길어지면 LLM은 이전에 자신이 한 말을 "맞다"고 가정하고, 같은 방식으로 반복 시도한다.

**방어:**
- 같은 접근이 2번 실패하면 **접근 자체를 의심**한다. 3번째 시도는 금지
- 실패 시 "왜 실패했는가"를 먼저 진단한다
- 에러 메시지를 읽고, 가정을 검증한다

### 3.3 유령 수정 방어 (Phantom Fix)
"수정했습니다"라고 말하지만 실제로는 문제가 해결되지 않은 상태.

**방어:**
- 수정 후 반드시 make test 실행. "논리적으로 맞을 것"을 신뢰하지 않는다
- 테스트가 통과하더라도 의도한 동작이 실제로 되는지 Playwright로 확인한다

### 3.4 스코프 팽창 방어 (Scope Creep)
요청받은 것 이상을 "개선"하려는 경향.

**방어:**
- **Issue 1개 (또는 의존 관계 묶음) = PR 1개**
- "이것도 같이 하면 좋겠다"는 하지 않는다. 별도 이슈를 만든다
- 커밋 메시지에 이슈 번호를 명시한다

### 3.5 숫자 전파 방어 (Stale Number Propagation)
한 곳의 숫자를 바꾸고 다른 참조를 업데이트하지 않는 것.

**방어:**
- 숫자 변경 시 grep -ri "이전값"으로 전수 검색
- CLAUDE.md, README.md, STRATEGY.md 모두 확인

### 3.6 테스트 환각 방어 (Test Illusion)
테스트가 통과하지만 실제로는 타겟 코드를 실행하지 않는 경우.

**방어:**
- pytest-asyncio fixture는 반드시 @pytest_asyncio.fixture 사용
- asyncio_mode = "auto" 확인
- 조건부 로직 안에 핵심 assertion을 넣지 않는다

---

## 4. 하네스 원칙 요약

1. 모르면 읽는다         — 가정하지 않는다
2. 2번 실패하면 접근을 바꾼다  — 같은 시도 3회 금지
3. 수정 후 실행한다       — 논리적 확신을 신뢰하지 않는다
4. 스코프를 지킨다       — 요청된 것만 한다
5. 숫자를 grep한다       — 한 곳만 고치지 않는다
6. 시스템이 차단한다      — CI/test가 강제한다
7. 스크린샷으로 증명한다   — "동작할 것이다"는 증거가 아니다

## 5. 개발 품질 기준

### 테스트
| 항목 | 기준 |
|------|------|
| pytest | 13개 이상 (CRUD × happy/error/edge) |
| CI 통과 | 필수 |
| Playwright 캡처 | PR마다 스크린샷 첨부 |

### 코드
| 항목 | 기준 |
|------|------|
| docstring | 모든 함수 필수 |
| 커밋 | feat:/fix:/docs:/test:/chore: + 한국어 + 이슈 번호 |
| PR 단위 | Issue 묶음 = PR 1개 |

### 보안
| 항목 | 기준 |
|------|------|
| SQL Injection | 파라미터 바인딩 ? 사용 (f-string SQL 금지) |
| XSS | textContent 사용 (innerHTML 금지, 정적 요소 제외) |
| 입력 검증 | Pydantic 모델에서 서버 측 검증 |
